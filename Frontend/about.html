01 — Foundation

Memory Management:
The Core Problem

Every program needs memory to store data. In languages like C and C++, the programmer manually allocates memory using malloc() or new, and must free it using free() or delete. This model is powerful but dangerous — forget to free memory and you have a memory leak; free it too early and you have a dangling pointer; free it twice and you have undefined behavior.

Garbage collection (GC) is the automated solution. A runtime system tracks which objects are still in use and automatically reclaims those that aren't. The programmer is freed from manual memory management, but at the cost of some runtime overhead.

🧠 Key Concepts
Heap — Dynamic memory region where objects live
Stack — Function-local variables, auto-managed
GC Root — Starting point for reachability analysis (stack vars, globals)
Reachability — An object is alive if it can be reached from a GC root via references
Liveness — Only reachable objects are considered "live" and retained

02 — Algorithm One

Reference Counting

Incremental No Cycle Detection Immediate Reclamation

How It Works

Reference counting is the simplest GC strategy. Every object on the heap stores an integer — its reference count — representing how many pointers currently point to it.

  • When a new reference to an object is created, its count is incremented
  • When a reference is destroyed (goes out of scope, is reassigned), the count is decremented
  • When the count reaches zero, no references exist and the object is immediately freed
  • Freeing an object may decrement counts of objects it references, potentially cascading
Pseudocode — Reference Counting
function new_reference(obj):
    obj.ref_count += 1

function delete_reference(obj):
    obj.ref_count -= 1
    if obj.ref_count == 0:
        for ref in obj.references:
            delete_reference(ref)
        free(obj)  // immediate!

// Example:
a = allocate()   // a.ref_count = 1
b = a            // a.ref_count = 2
del b            // a.ref_count = 1
del a            // a.ref_count = 0 → freed!
⚠️ The Cycle Problem

Reference counting's fatal flaw: cyclic references. If Object A references B and B references A, both have a ref count of 1 even when no external pointers exist. Neither will ever be freed — this is a memory leak.

Solutions include: a separate cycle detector (CPython's approach), weak references for back-pointers (Swift, Rust), or combining with mark-and-sweep for cycle collection.

03 — Algorithm Two

Mark & Sweep

Stop-the-World Handles Cycles Complete Collection

Mark & Sweep is the foundational GC algorithm that most modern collectors build upon. It operates in two clearly distinct phases and correctly handles all reachability scenarios — including cyclic references.

Phase 1
🔍 Mark

Starting from every GC root (stack variables, global variables, registers), the collector traverses the entire object graph using depth-first or breadth-first search. Every reachable object gets its mark bit set.

This is essentially graph traversal — the GC treats the heap as a directed graph where edges are references.
Phase 2
🧹 Sweep

The collector linearly scans every object in the heap. Objects with their mark bit set (reachable) have their bit cleared for next cycle. Objects with bit clear (unreachable) are freed.

Sweep time is proportional to heap size, not live object count — it must inspect everything.
Pseudocode — Mark & Sweep
function mark_and_sweep():
    // PHASE 1: Mark
    worklist = get_all_roots()
    while worklist not empty:
        obj = worklist.pop()
        if not obj.marked:
            obj.marked = true
            for ref in obj.references:
                worklist.push(ref)

    // PHASE 2: Sweep
    for obj in all_heap_objects:
        if obj.marked:
            obj.marked = false  // reset for next GC
        else:
            free(obj)
⚡ Variants & Optimizations
Tri-color marking — White (unvisited), Gray (in progress), Black (done) — enables incremental GC
Mark-Compact — After sweeping, compact live objects to eliminate fragmentation
Mark-Copy — Copy live objects to a new region instead of freeing in-place
Incremental marking — Spread mark phase across multiple small pauses
Concurrent GC — Run GC threads alongside application threads (requires write barriers)
Real-world Usage
Go (tricolor concurrent) Early JVM JavaScript engines (V8 major GC) Lua Ruby (CRuby)
04 — Algorithm Three

Generational GC

High Throughput Short Minor Pauses Complex Implementation
💡
The Generational Hypothesis

"Most objects die young." Empirical studies show that the vast majority of objects allocated in typical programs become unreachable very quickly — often within milliseconds. A small percentage (long-lived caches, singletons, etc.) survive for the entire program duration.

How It Works

Generational GC divides the heap into generations based on object age. New objects are allocated in the young generation (also called nursery). Objects that survive multiple GC cycles are promoted to older generations.

Because most objects die young, collecting only the young generation (minor GC) is fast and efficient — you process a small region and collect a lot of garbage. The old generation is collected less frequently (major GC).

  • Young Gen: Small, collected frequently (milliseconds)
  • Old Gen: Large, collected rarely (seconds)
  • Promotion: Objects surviving N minor GCs move to old gen
  • Write Barriers: Track old→young references for minor GC correctness

🔗 Write Barriers — The Secret Sauce

Minor GC must find all references from old objects into the young generation (cross-generational references). Without tracking these, live young objects referenced only by old objects would be incorrectly freed.

Write barriers intercept every pointer store. When an old object writes a reference to a young object, that old object is recorded in a remembered set or card table. Minor GC treats these as additional roots.

05 — Analysis

Trade-offs &
Performance

⏱️

Latency vs Throughput

Reference counting has low latency (no pauses) but lower throughput (constant overhead per pointer write). Mark & Sweep has high throughput but can cause large pauses. Generational GC optimizes both with short minor pauses and infrequent major pauses.

💾

Memory Overhead

GC systems typically need 2–5× more memory than live data to work efficiently (to avoid frequent collections). Reference counting adds one integer per object. Generational GC needs extra memory for survivor spaces and remembered sets.

🔄

Fragmentation

Mark & Sweep can fragment memory as objects are freed in-place. Mark-Compact and generational collectors using copying avoid fragmentation by relocating live objects, but require updating all references — a significant challenge.

🧵

Concurrency

Modern GCs aim to run concurrently with the application. This requires write barriers (intercept pointer stores) and read barriers to ensure the GC's view of the heap stays consistent. Go's GC and JVM's ZGC are examples of highly concurrent collectors.

06 — Real World

Production GC Systems

☕
JVM (Java/Kotlin/Scala)
G1GC, ZGC, Shenandoah

The JVM offers multiple GC options. G1GC (Garbage First) divides heap into equal-sized regions and prioritizes collecting regions with most garbage. ZGC targets sub-millisecond pauses even on terabyte heaps using colored pointers and load barriers.

🦎
V8 (JavaScript)
Orinoco — Generational + Concurrent

V8's Orinoco GC uses a generational approach. Minor GC (Scavenge) uses a copying collector on the young generation. Major GC uses incremental marking and concurrent sweeping to minimize pause times in the browser and Node.js.

🐍
CPython
Reference Counting + Cycle Detector

Python primarily uses reference counting for immediate deallocation. A supplemental cycle-detecting GC runs periodically to collect cyclic garbage that reference counting cannot handle. Objects are sorted into 3 generations for the cycle detector.

🦫
Go
Tricolor Concurrent Mark & Sweep

Go uses a concurrent, tricolor mark-and-sweep collector. It aims for sub-millisecond stop-the-world pauses by doing most work concurrently with the application. Write barriers maintain invariants during concurrent marking.

🔷
.NET CLR
Generational + Server/Workstation modes

.NET uses a generational GC with two modes: Workstation GC (low latency, single-threaded) and Server GC (high throughput, multi-threaded with per-core heaps). Background GC allows major collections to run concurrently.

🦅
Swift / Obj-C
Automatic Reference Counting (ARC)

Swift uses ARC, which is compile-time reference counting — the compiler inserts retain/release calls automatically. Unlike runtime GC, this has zero pause times. Weak and unowned references handle cycles. ARC is deterministic and cache-friendly.
